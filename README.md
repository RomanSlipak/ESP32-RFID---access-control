## Опис проекту:
Система контролю доступу на базі ESP32 дозволяє керувати доступом до приміщення за допомогою RFID-карток та брелоків стандарту MIFARE. Авторизовані картки зберігаються на SD-карті, що дозволяє системі повноцінно працювати навіть без підключення до мережі.

Telegram-бот забезпечує зручне дистанційне управління: адміністратор отримує сповіщення про кожну спробу доступу та може додати або відхилити нову картку прямо з телефону. TFT-дисплей та зуммер забезпечують локальний зворотній зв'язок.

# Можливості
 
- 🔐 **RFID-аутентифікація** - зчитування карток/брелоків MIFARE (MFRC522)
- 📩 **Telegram-сповіщення** - миттєве повідомлення про кожну спробу доступу
- 📡 **Управління картками через Telegram** - кнопки «Додати» / «Відмовити» прямо в чаті
- 💾 **Зберігання на SD-карті** - база авторизованих карток у JSON-форматі
- 🖥️ **TFT-дисплей (ST7789)** - відображення статусу та імені користувача
- 🔊 **Зуммер** - звуковий сигнал успіху або відмови
- 📂 **Автономний режим** - система продовжує працювати при відсутності Wi-Fi

### Схема підключення
Усі модулі підключаються до ESP32 через SPI. MFRC522, ST7789 та SD-модуль **поділяють спільну SPI-шину** (MOSI, MISO, SCK), кожен має окремий сигнал вибору (CS).

#### SPI-шина (спільна для RFID, TFT, SD)
 
| ESP32 GPIO | Сигнал SPI |
|------------|-----------|
| GPIO 18    | SCK       |
| GPIO 23    | MOSI      |
| GPIO 19    | MISO      |
 
#### RFID-зчитувач MFRC522
 
| MFRC522 | ESP32 GPIO | Опис |
|---------|-----------|------|
| SDA (SS/CS) | GPIO 5  | Chip Select |
| SCK         | GPIO 18 | SPI Clock (спільний) |
| MOSI        | GPIO 23 | SPI Data (спільний) |
| MISO        | GPIO 19 | SPI Data (спільний) |
| RST         | GPIO 27 | Reset |
| 3.3V        | 3.3V    | Живлення |
| GND         | GND     | Земля |
 
#### TFT-дисплей ST7789
 
| ST7789 | ESP32 GPIO | Опис |
|--------|-----------|------|
| CS  | GPIO 15 | Chip Select |
| DC  | GPIO 2  | Data/Command |
| RST | GPIO 16 | Reset |
| SDA (MOSI) | GPIO 23 | SPI Data (спільний) |
| SCL (SCK)  | GPIO 18 | SPI Clock (спільний) |
| BLK / LED  | 3.3V або GPIO | Підсвітка |
| VCC | 3.3V | Живлення |
| GND | GND  | Земля |
 
#### SD-модуль
 
| SD-модуль | ESP32 GPIO | Опис |
|-----------|-----------|------|
| CS   | GPIO 17 | Chip Select |
| MOSI | GPIO 23 | SPI Data (спільний) |
| SCK  | GPIO 18 | SPI Clock (спільний) |
| MISO | GPIO 19 | SPI Data (спільний) |
| VCC  | 3.3V або 5V | Живлення (залежно від модуля) |
| GND  | GND | Земля |
 
#### Зуммер та реле
 
| Компонент | ESP32 GPIO | Опис |
|-----------|-----------|------|
| Зуммер (+) | GPIO 21 | Сигнал управління |
| Зуммер (–) | GND     | Земля |
| Реле IN    | GPIO 22 | Сигнал управління |
| Реле VCC   | 5V      | Живлення |
| Реле GND   | GND     | Земля |

# Останні кроки
Тестування перед початком роботи:
  1. Тест №1 - Перевірка Serial Monitor. Відкрити Tools → Serial Monitor, встановити відповідну до Serial.begin швидкість. Після завантаження прошивки Serial Monitor повинен відображати послідовні повідомлення ініціалізації: "[BUZZER] OK", "[TFT] OK", "[RFID] Версія: 0x92", "[SD] OK", "[WiFi] OK IP: XXX.XXX.X.X".
  2. Тест №2 - Перевірка RFID. Піднести будь-яку картку MIFARE до зчитувача. У Serial Monitor повинен з'явитись рядок "[RFID] UID: A1B2C3D4" (конкретне значення залежить від картки). Якщо UID не з'являється, перевірити підключення пінів RFID і CS.
  3. Тест №3 - Перевірка Telegram. Відкрити бота у Telegram і відправити /status. Бот повинен відповісти повідомленням зі статусом системи. При піднесенні невідомої картки у Telegram повинно з'явитись повідомлення з кнопками «Додати» і «Відмовити».
  4. Тест №4 - Повний цикл доступу. Натиснути «Додати» у Telegram, ввести ім'я, піднести ту саму картку. Система повинна відобразити ім'я на дисплеї, відтворити сигнал успіху, увімкнути реле і надіслати повідомлення про доступ у Telegram.
  5. Тест №5 - Перевірка автономного режиму. Відключити Wi-Fi (або тимчасово ввести неправильний SSID). Перезавантажити систему. Переконатись, що RFID-автентифікація продовжує працювати коректно, хоча Telegram-сповіщення не надходять.

## Основні файли:
# Access_control
Назва файлу: access_control.ino

Головний файл прошивки. Містить `setup()` та `loop()`:
- ініціалізація всіх модулів (RFID, TFT, SD, Wi-Fi, Telegram, зуммер),
- основний цикл опитування RFID-зчитувача,
- логіка перевірки картки (дозволити / відхилити),
- обробка вхідних повідомлень Telegram-бота,
- управління реле та дисплеєм за результатом перевірки.

---

### Config
Назва файлу: config.h

Файл конфігурації. Централізоване місце для всіх налаштувань:
- константи Wi-Fi (SSID, пароль),
- токен та Chat ID Telegram-бота,
- номери GPIO для всіх підключених модулів (RFID, TFT, SD, зуммер, реле),
- інші параметри (час утримання реле, тайм-аут Wi-Fi тощо).
  
> Усі налаштування змінюються лише тут, решта файлів підтягують їх через `#include "config.h"`.

---

### Buzzer
Назва файлу: buzzer.h

Модуль управління зуммером. Надає функції для відтворення звукових сигналів:
- сигнал **успіху** (доступ дозволено),
- сигнал **відмови** (картка не знайдена або відхилена),
- сигнал **завантаження** (ініціалізація системи).

---

### Display
Назва файлу: display.h

Модуль TFT-дисплея (ST7789). Відповідає за:
- ініціалізацію дисплея,
- відображення стартового екрана,
- виведення статусних повідомлень («Готово», «Доступ дозволено», «Доступ заборонено»),
- відображення імені авторизованого користувача.

---

### RFID
Назва файлу: rfid.h

Модуль RFID-зчитувача (MFRC522). Відповідає за:
- ініціалізацію модуля та перевірку версії чіпа,
- зчитування UID картки / брелока,
- перетворення UID у рядок для порівняння з базою,
- звільнення карти після зчитування.

---

### Storage
Назва файлу: storage.h

Модуль роботи з SD-карткою. Відповідає за:
- ініціалізацію SD та перевірку наявності файлу бази карток,
- зчитування JSON-файлу авторизованих карток,
- пошук UID у базі та повернення імені користувача,
- збереження нової картки до бази (за командою з Telegram),
- ведення журналу подій (лог доступу).

---

### Telegram
Назва файлу: telegram.h

Модуль Telegram-бота (UniversalTelegramBot). Відповідає за:
- ініціалізацію бота та перевірку підключення,
- відправлення сповіщень про спроби доступу (UID, ім'я, час),
- відправлення inline-кнопок **«Додати»** / **«Відмовити»** для невідомих карток,
- обробку callback-запитів від кнопок,
- обробку команди `/status` для отримання поточного стану системи.

## Команди Telegram-бота

Бот приймає як текстові команди, так і натискання inline-кнопок, які надсилаються автоматично при піднесенні нової картки.
 
> [!NOTE]
> Бот реагує **лише** на повідомлення від Chat ID, зазначеного в `config.h`. Усі інші відправники ігноруються.

### Текстові команди
 
| Команда | Синтаксис | Опис |
|---------|-----------|------|
| `/start` | `/start` | Привітальне повідомлення та перелік доступних команд |
| `/status` | `/status` | Поточний стан системи: Wi-Fi сигнал, IP-адреса, час роботи (uptime), стан SD-карти, час останнього проходу |
| `/list` | `/list` | Список усіх авторизованих карток у форматі `UID — Ім'я` |
| `/remove` | `/remove `UID | Видалити картку з бази за її UID |
| `/open` | `/open` | Дистанційно відкрити замок (активувати реле на заданий час) |
| `/help` | `/help` | Повторно надіслати перелік команд |
